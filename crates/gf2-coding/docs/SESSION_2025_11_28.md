## Design Decisions

### 1. gf2-core Stays Serial
**Decision**: Keep gf2-core serial, parallelism lives in gf2-coding  
**Rationale**:
- Clean API boundary (caller controls parallelism strategy)
- No hidden thread pools or synchronization overhead
- Composability (block-level vs. data-level parallelism)
- Predictable performance characteristics

### 2. Batch-First API Design
**Decision**: Always design for batches (even if batch size = 1)  
**Rationale**:
- Prepares for GPU/FPGA (amortizes transfer overhead)
- Natural fit for SDR workloads (process frames in batches)
- Enables future optimizations without API changes

### 3. Sequential Encoder (For Now)
**Decision**: Keep encoder sequential until Sync bounds resolved  
**Rationale**:
- Encoder uses cached generator matrices with Arc
- Sync bounds need careful review (gf2-core types)
- Decoder is the bottleneck (37× gap vs. 8.2× for encoder)
- Prioritize decoding parallelization first

### 4. Thread-Local Decoder Instances
**Decision**: Each rayon thread creates its own decoder  
**Rationale**:
- Avoids Sync complexity (decoders have mutable state)
- Simple implementation (no Arc<Mutex<>> needed)
- Scales naturally with core count
- Code cloning is cheap (Arc for large matrices)

## Key Insights

### 1. TDD Pays Off
- Writing tests first revealed state leakage bug
- Bug would have caused incorrect results in production
- Tests act as specification and regression protection

### 2. Rayon Just Works™
- Minimal code changes for parallelization
- Automatic work stealing and load balancing
- Good scaling on 24-core CPU (6.7× for decoder)

### 3. Memory Bandwidth Bottleneck
- Not hitting full 24× speedup (only 6.7×)
- Likely memory bandwidth bound (sparse matrix access)
- SIMD will help by reducing memory traffic

### 4. API Design Matters
- Batch-first design enables future optimizations
- Clean separation: gf2-core (serial) vs. gf2-coding (parallel)
- Easy to add GPU backend later (ComputeBackend trait)
