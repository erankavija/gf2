# Lean4 Formal Verification Pipeline

Comprehensive documentation for the Charon/Aeneas → Lean4 verification
infrastructure for `gf2-core` field arithmetic.

## Overview

We translate Rust source code from `gf2-core` into Lean4 for formal
verification using:

1. **Charon** — Compiles Rust to LLBC (Low-Level Borrow Calculus), an
   intermediate representation.
2. **Aeneas** — Translates LLBC to pure Lean4 code.
3. **Lake** — Lean4 build system that type-checks the generated code.

The pipeline is automated via `scripts/verify-lean.sh`.

### What is verified

| Module | Status | Notes |
|--------|--------|-------|
| `gfp/` (prime field `Fp<P>`) | Verified | Montgomery multiplication, field ops, const generics |
| `gfpn/` (`QuadraticExt`, `CubicExt`) | Verified | Tower extensions over `ExtConfig` trait |
| `field/traits` | Extracted (opaque bodies) | `FiniteField`, `ConstField` trait declarations |
| Everything else | Excluded | Out of scope (bitvec, matrix, SIMD, etc.) |

### Generated files

| File | Contents |
|------|----------|
| `proofs/Gf2Core/Types.lean` | Type definitions (`Fp<P>`, `MontConsts<P>`, `QuadraticExt`, `CubicExt`, trait structs) |
| `proofs/Gf2Core/Funs.lean` | Function bodies (Montgomery arithmetic, extension field operations) |
| `proofs/Gf2Core/FunsExternal.lean` | Axioms for external/opaque functions (std library ops) |
| `proofs/Gf2Core/FunsExternal_Template.lean` | Template generated by Aeneas (source of truth for FunsExternal) |

## Pipeline Stages

### Running the full pipeline

```bash
./scripts/verify-lean.sh
```

Or just build the committed Lean files (no Charon/Aeneas required):

```bash
cd proofs && lake build
```

### Stage 1: Charon Extraction

```bash
charon cargo \
  --preset aeneas \
  --opaque 'gf2_core::field' \
  --opaque 'gf2_core::gf2m' \
  --opaque 'gf2_core::bitvec' \
  --opaque 'gf2_core::bitslice' \
  --opaque 'gf2_core::matrix' \
  --opaque 'gf2_core::sparse' \
  --opaque 'gf2_core::alg' \
  --opaque 'gf2_core::compute' \
  --opaque 'gf2_core::kernels' \
  --opaque 'gf2_core::primitive_polys' \
  --opaque 'gf2_core::io' \
  --opaque 'gf2_core::macros' \
  --dest-file target/charon/gf2_core.llbc \
  -- --manifest-path crates/gf2-core/Cargo.toml --no-default-features
```

**Transparent modules** (fully extracted): `gfp/`, `gfpn/`

**Opaque modules** (declarations only, no bodies):

| Module | Why opaque |
|--------|------------|
| `field` | Trait declarations extracted; bodies contain HRTB `for<'a>` bounds |
| `gf2m` | Runtime field parameters, `Vec<u64>` storage |
| `bitvec`, `bitslice`, `matrix`, `sparse`, `alg` | Out of scope |
| `compute` | Rayon parallelism, unsupported by Aeneas |
| `kernels` | SIMD dispatch, unsupported by Aeneas |
| `primitive_polys` | Static data tables |
| `io`, `macros` | Serde / proc macros, unsupported |

**Expected warnings** (benign):
- 13 "Type error after transformations: Mismatched type generics" for
  `CubicExt`/`QuadraticExt` — Charon's `required_extra_params` generates
  duplicate entries (expected 4 generic args, got 7). Aeneas handles this
  correctly via Lean4 implicit argument inference.
- 1 "Type error after translation: Found incorrect clause var" for `ExtConfig`
  trait — also benign.

### Stage 2: Aeneas Translation

```bash
aeneas -backend lean -dest proofs/Gf2Core -split-files target/charon/gf2_core.llbc
```

Translates LLBC to Lean4. Generates `Types.lean`, `Funs.lean`, and
`FunsExternal_Template.lean`.

### Stage 3: Post-processing

Two automated fixups run after Aeneas:

#### 3a. Duplicate field dedup (`scripts/fix-aeneas-dupes.py`)

**Problem**: The `FiniteField` trait has supertype bounds on three associated
types (`Self`, `Characteristic`, `Wide`), each requiring `Clone`, `Eq`, etc.
Aeneas generates duplicate field names (e.g., three fields all named
`corecloneCloneInst`).

**Fix**: The script detects FiniteField struct/instance blocks and renames
duplicates with type suffixes:
- First occurrence → kept as-is (Self group)
- Second occurrence → `CharacteristicInst` suffix
- Third occurrence → `WideInst` suffix

The script handles two structural patterns:
- **gfp** (single-line): `field.traits.FiniteField` and `:= {` on the same line
- **gfpn** (multi-line): `field.traits.FiniteField` and `:= {` separated by
  several lines — detected via a 10-line lookback window

The script also fixes **projection paths** in value continuation lines. When a
field like `corecloneCloneInst` is renamed to `corecloneCloneCharacteristicInst`,
any projection like `.FiniteFieldInst.corecloneCloneInst` on the following
value line is also updated to `.FiniteFieldInst.corecloneCloneCharacteristicInst`.

#### 3b. FunsExternal.lean regeneration

```bash
cp proofs/Gf2Core/FunsExternal_Template.lean proofs/Gf2Core/FunsExternal.lean
```

Copied from the template, then customized. Current customizations:
- `core.option.Option.map`: Replaced axiom with a concrete `def` matching
  Rust's `Option::map` semantics (`some t, f => call f t; ok (some r)` /
  `none, _ => ok none`). Required for extension field inversion proofs.

Remaining axioms: `Option.expect`, `Try.branch`, `FromResidual.from_residual`,
`assert_receiver_is_total_eq`.

### Stage 4: Lake Build

```bash
cd proofs && lake build
```

Type-checks all generated Lean4 code (~1594 build jobs). Only expected warnings
are `sorry` in the Aeneas standard library (upstream).

## Charon Patches

Three patches applied to `expand_associated_types.rs` in the local Charon build
at `/data/aeneas-build/charon/`. All are in the file:
`charon/src/transform/normalize/expand_associated_types.rs`

### 1. HRTB Erase (commit `752460d1`)

**Problem**: When expanding associated types, Charon's `move_from_under_binder()`
panicked on Higher-Ranked Trait Bounds (`for<'a> ...`). The HRTB region binder
introduced a binding level mismatch.

**Fix**: Changed 3 call sites from `move_from_under_binder()` to `.erase()`,
which erases the binder entirely. Safe because the erased regions are only used
for lifetime checking, which is already complete at this stage.

**Test**: `charon/tests/ui/hrtb-associated-types.rs`

### 2. SelfClause/Local(ZERO) Unification (commit `5dde2319`)

**Problem**: Charon represents `Self` in trait contexts as both `SelfClause` and
`Local(ZERO)` in different codepaths. When resolving associated types in trait
default method implementations, `lookup_type_replacement()` failed to find
replacements because it searched for `SelfClause` but the replacement map was
keyed by `Local(ZERO)` (or vice versa).

**Fix**: Added bidirectional fallback in `lookup_type_replacement()`: if the
primary lookup fails, try the other representation. Also extended
`iter_self_paths()` to yield both `SelfClause` and `Local(ZERO)` variants.

**Test**: `charon/tests/ui/trait-default-method-assoc-type.rs`

### 3. Implied Clause Constraint Propagation (commit `18ca0d00`)

**Problem**: When computing trait modifications in `compute_trait_modifications()`,
the code only propagated `type_constraints` from direct `parent_clauses` but not
from `implied_clauses`. For traits like `ConstField: FiniteField` where the
constraint on associated types flows through an implied clause, this caused 86
"Could not compute the value of an associated type" warnings and incomplete
type resolution.

**Fix**: Added propagation of `type_constraints` from `tr.implied_clauses` in
`compute_trait_modifications()`, using the same logic as for `parent_clauses`.

**Test**: `charon/tests/ui/implied-clause-assoc-type-constraint.rs`

All 278 Charon UI tests pass with these patches.

## Const Generics

`Fp<const P: u64>` extracts correctly without needing monomorphization wrappers.
Charon handles const generics natively and Aeneas translates `P` as a Lean4
parameter `(P : Std.U64)`.

## Regenerating After Rust Changes

If Rust source in `gfp/` or `gfpn/` changes:

```bash
./scripts/verify-lean.sh
git add proofs/Gf2Core/
```

## Troubleshooting

### "Could not compute the value of an associated type"

If this warning reappears after Charon updates, it likely means the implied
clause propagation patch was lost. Reapply patch 3.

### "field has already been specified" in lake build

The dedup script (`fix-aeneas-dupes.py`) failed to detect a FiniteField block.
Check whether the Aeneas output format changed (e.g., the `:= {` line moved
further from the `field.traits.FiniteField` reference — increase the lookback
window from 10 if needed).

### Type mismatch on Characteristic/Wide fields

The projection path fixup in `fix-aeneas-dupes.py` didn't fire. Ensure the
script is processing fresh Aeneas output (not already-processed files). The
pipeline script handles this correctly, but manual re-runs of the dedup script
on already-processed files will silently skip the fixup.
